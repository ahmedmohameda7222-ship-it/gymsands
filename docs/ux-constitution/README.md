# Plaivra UX Constitution

**Version:** 2026.1  
**Status:** Source of truth for UI/UX decisions  
**Product direction:** Premium-feeling mobile-first fitness, nutrition, wellness, and AI-assisted tracking app.  
**Important:** Plaivra should feel high-quality and deliberate. It should not copy Apple UI visually, but it should meet the same level of clarity, restraint, touch comfort, speed, trust, and polish expected from premium consumer software.

---

## 1. Purpose

This document is the permanent UI/UX reference for Plaivra.

Use it before changing buttons, spacing, navigation, cards, action placement, loading states, AI request flows, or mobile behavior.

The goal is to prevent endless subjective redesign cycles. A screen is considered ready only when it satisfies this constitution or has a clearly documented exception.

---

## 2. Source hierarchy

Plaivra does not follow many references equally. Plaivra has one internal standard, informed by several external sources.

Decision priority:

1. **Accessibility and safety** — WCAG and platform accessibility rules.
2. **Touch comfort** — Apple HIG, Android/Material accessibility, and touch ergonomics research.
3. **Human behavior** — perceived speed, feedback, user control, and error recovery.
4. **Plaivra product logic** — fitness/nutrition/wellness workflows, subscription-readiness, and AI trust.
5. **Plaivra design system** — spacing, density, radius, button variants, typography, and motion.
6. **Visual taste** — colors, gradients, shadows, glass, decorative treatment.

If visual taste conflicts with usability, accessibility, or product clarity, visual taste loses.

## 4. Plaivra product principles

### 4.1 Premium means restraint

Premium does not mean more cards, more gradients, more shadows, or more visible buttons.

Premium means:

- Fewer visible decisions.
- Clear hierarchy.
- Strong spacing consistency.
- Fast feedback.
- Comfortable touch targets.
- Calm empty, loading, and error states.
- No silent destructive or AI-driven changes.
- No screens that look generated by assembling random components.

### 4.2 The user should understand the screen in two seconds

Every screen must answer:

1. Where am I?
2. What is the main thing I can do here?
3. What changed after I tapped?
4. Is anything still saving, syncing, or waiting?

If a user needs to read many buttons to understand the screen, the hierarchy is failing.

### 4.3 One primary action

A visible screen section should normally have only one primary action.

Allowed:

- One primary CTA.
- Up to two nearby secondary actions if genuinely needed.
- Rare, administrative, or dangerous actions inside a More menu, edit mode, or separate settings subpage.

Not allowed:

- Multiple primary buttons competing in the same card or screen section.
- Important and unimportant actions using the same visual weight.
- Destructive actions placed next to normal primary actions without separation.

## 5. Button and touch standards

### 5.1 Minimum touch size

All tappable controls must have a minimum effective hit area of **48 x 48 px**.

The visible icon can be smaller, but the tappable area must not be smaller.

| Element | Standard |
|---|---:|
| Minimum tappable area | 48 x 48 px |
| Icon-only button visual icon | 20-24 px |
| Normal button height | 44-48 px |
| Primary mobile CTA height | 52-56 px |
| Sticky bottom CTA height | 52-56 px plus safe-area bottom |
| Clickable settings/list row | 56-64 px |
| Destructive action | Visible label or protected menu/sheet action |

### 5.2 Button hierarchy

All buttons must be classified as one of these:

| Type | Usage |
|---|---|
| Primary | Main next step for the current screen or section. |
| Secondary | Useful but not the main action. |
| Tertiary / Ghost | Low-emphasis helper action. |
| Icon-only | Repeated, familiar, low-risk action with accessible label. |
| Destructive | Delete, remove, revoke, cancel, or irreversible action. |
| More menu item | Rare, administrative, overflow, or advanced action. |
| Edit-mode action | Action that should only appear after entering edit mode. |

### 5.3 Button placement

| Situation | Placement rule |
|---|---|
| Action affects one card | Keep inside that card. |
| Action affects the whole page | Use page header action or mobile sticky CTA. |
| Fast repeated action | Place close to the relevant item. |
| Destructive action | Separate visually and require confirmation or undo. |
| Rare action | Move to More menu. |
| Complex action | Use a full page or focused flow. |
| Quick contextual action | Use sheet/dialog only when it reduces page clutter. |
| AI action | Show only when context is clear and user approval remains explicit. |

## 6. Spacing standards

Plaivra uses a fixed spacing scale. Avoid arbitrary spacing values.

Approved spacing scale:

```txt
2px  = hairline / tiny optical adjustment
4px  = very tight relation
6px  = label/input micro spacing
8px  = related items / button group gap
12px = compact comfortable gap
16px = default content gap
20px = premium breathing gap
24px = section/card internal separation
32px = major section separation
40px = page-level separation
48px = large rhythm / hero rhythm
64px = major layout rhythm
```

### 6.1 Practical spacing rules

| Relationship | Standard |
|---|---:|
| Label to input | 6-8 px |
| Input to helper/error text | 4-8 px |
| Short text to button | 16 px |
| Paragraph to primary CTA | 20-24 px |
| Hero copy to CTA | 24-32 px |
| Card content to action row | 16-20 px |
| Buttons in same group | 8-12 px |
| Stacked mobile buttons | 8-12 px |
| Unrelated action groups | 24-32 px |
| Between cards on mobile | 16 px |
| Between cards on desktop | 20-24 px |
| Between major sections | 32-40 px |

### 6.2 Forbidden spacing behavior

Avoid random values such as 13 px, 17 px, 29 px unless there is a specific optical correction. If spacing is repeated, it must become a token or reuse the approved scale.

## 7. Feedback and speed standards

### 7.1 Tap feedback

Every tap must produce visible feedback immediately.

Target:

- Press/active visual feedback under 100 ms.
- Safe optimistic state change for repeated low-risk actions.
- Pending state for saves, sync, imports, exports, and network operations.

### 7.2 Optimistic UI

Use optimistic UI for actions where immediate response matters and failure can be safely reverted.

Good optimistic candidates:

- Mark habit done.
- Mark daily task done.
- Mark supplement taken.
- Log water.
- Mark meal done.
- Toggle favorite.
- Save small preference.
- Like/check-style actions.

If the server fails:

1. Revert the local state.
2. Show a clear toast or inline error.
3. Do not leave the user uncertain about whether the action worked.

### 7.3 Do not use optimistic UI for high-risk actions

Do not use blind optimistic UI for:

- Delete account.
- Delete workout plan permanently.
- Cancel subscription.
- Payments.
- Revoke permissions.
- Major plan overwrite.
- Bulk delete.
- Medical/safety-sensitive actions.
- Any action that cannot be easily reversed.

Use confirmation, review, undo, or a clearly pending state.

## 8. Loading, error, empty, and offline states

Every major screen must support:

- Loading state.
- Empty state.
- Error state.
- Retry or recovery action.
- Clear offline/sync behavior where relevant.

### 8.1 Loading rules

| Time | Required behavior |
|---|---|
| Under 1 second | Subtle pending state is enough. |
| 1-10 seconds | Skeleton/spinner plus contextual message. |
| Over 10 seconds | Stronger progress/status explanation and recovery path. |

### 8.2 Empty state rules

A good empty state includes:

- What is empty.
- Why it matters.
- The next best action.

Bad empty state:

```txt
No data.
```

Good empty state:

```txt
No meals planned for today.
Add your first planned meal or import a plan you already approved.
```

### 8.3 Error state rules

Every error must include:

- Human-readable message.
- Retry or fallback action.
- No raw technical stack traces for normal users.
- Optional technical details only when useful for debugging.

## 9. AI action standards

Plaivra can prepare ChatGPT requests and store approved results. It must not create confusion about whether Plaivra is changing data automatically.

AI action rules:

1. AI actions must be contextual, not sprayed across every screen.
2. The button label must explain the goal, not just say "Ask AI".
3. The request must summarize what context will be used.
4. No data should change silently after an AI recommendation.
5. The user must approve meaningful changes before they are saved.
6. Permission requirements must be clear before requesting or granting access.
7. AI write access must remain revocable.

Preferred wording examples:

- "Ask ChatGPT to replace this exercise"
- "Ask ChatGPT to rebalance this week"
- "Ask ChatGPT for cheaper grocery swaps"

Avoid vague labels:

- "AI"
- "Generate"
- "Magic"
- "Fix it"

## 10. Navigation and information architecture

### 10.1 Mobile navigation

Mobile navigation should prioritize daily repeated behavior.

Primary mobile nav should include only the most important sections. Additional sections should live in More, Settings, or contextual hubs.

### 10.2 Route-level purpose

Each route must have one dominant job.

Examples:

| Route type | Expected dominant job |
|---|---|
| Dashboard | See today and take the next most likely action. |
| Workout plan | Start, view, and manage active training. |
| Workout session | Focus on logging the current workout only. |
| Calories | Log and review daily nutrition. |
| Meal plan | Plan meals and mark planned food done. |
| Progress | Add or review body/progress data. |
| Settings | Configure account, privacy, AI, preferences, and data controls. |

If a route tries to do too many unrelated jobs, split it, hide advanced actions, or move actions into contextual flows.

## 11. Premium visual treatment

Premium visual design is allowed only when it supports clarity.

### 11.1 Cards

Use cards to group related information. Do not wrap every row in a heavy card.

A card should have:

- A clear title or purpose.
- Related content.
- Related actions.
- Consistent padding.

### 11.2 Shadows, blur, gradients, and glass

Use decorative effects with restraint.

Allowed:

- Subtle elevation for important surfaces.
- Blur for navigation/sheets when it improves depth.
- Primary color for selected/active states and CTAs.

Avoid:

- Heavy shadows everywhere.
- Too many glass cards.
- Multiple competing gradients.
- Decorative effects that lower contrast or make text harder to read.

---

## 12. Motion standards

Motion should confirm cause and effect. It should not delay repeated daily actions.

| Interaction | Duration |
|---|---:|
| Button press feedback | 80-120 ms |
| Small UI reveal | 150-220 ms |
| Page transition | 180-260 ms |
| Sheet open/close | 250-350 ms |

Motion rules:

- Avoid excessive bounce.
- Avoid long animations on repeated logging actions.
- Respect reduced-motion settings where possible.
- Use motion to clarify hierarchy, not decorate randomly.

## 13. Subscription-readiness standard

Plaivra is intended to support subscriptions in the future. The UI must already feel trustworthy enough to pay for.

Subscription-ready UI means:

- No unfinished-looking flows.
- No vague buttons.
- No silent data changes.
- Clear permission boundaries.
- Clear privacy/data export/delete behavior.
- High-quality empty/loading/error states.
- Consistent spacing and action hierarchy.
- Clear upgrade/paywall surfaces when added later.

A user should feel the app is reliable before being asked to pay.

## 14. Audit decision labels

When auditing a screen or component, every button/action should receive one decision:

| Decision | Meaning |
|---|---|
| Keep | Correct as-is. |
| Resize | Hit area or visual height is wrong. |
| Move | Correct action, wrong location. |
| Restyle | Correct action, wrong hierarchy/variant. |
| Rename | Label is vague or misleading. |
| Hide | Should not be visible by default. |
| Move to menu | Rare/admin action should be overflow. |
| Move to edit mode | Should only appear after edit mode starts. |
| Convert to sheet | Better as focused contextual action. |
| Delete | Action is unnecessary or harmful. |
| Needs confirmation | Risk/destructive action lacks protection. |
| Needs feedback | Action lacks immediate feedback/loading/error state. |

## 15. Route audit checklist

Use this checklist for every route:

```txt
1. What is the primary job of this route?
2. What is the single primary action?
3. Are there more than two visible secondary actions near the primary action?
4. Are rare actions hidden in menu/sheet/edit mode?
5. Are destructive actions separated and protected?
6. Are all tappable controls at least 48 x 48 px effective hit area?
7. Is spacing using the Plaivra spacing scale?
8. Is the button close to the content it affects?
9. Does every tap have immediate feedback?
10. Is optimistic UI used where safe and avoided where risky?
11. Are loading, empty, error, retry, and offline states covered?
12. Does the screen feel calm and deliberate rather than assembled?
13. Would this screen feel acceptable in a paid subscription product?
```

---

## 16. Definition of done for UI/UX changes

A UI/UX change is not done when it looks nicer. It is done when:

- It satisfies this constitution.
- It improves or preserves the route's primary job.
- It reduces unnecessary visible decisions.
- It improves touch comfort and spacing consistency.
- It preserves accessibility.
- It includes feedback, loading, and error behavior where needed.
- It does not introduce silent destructive or AI-driven changes.
- It is documented in the UX progress tracker if part of a route audit.
